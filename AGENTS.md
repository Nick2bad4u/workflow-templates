# Workflow Templates Repository Instructions

## Repository Purpose

This repository publishes GitHub Actions workflow templates under `.github/workflow-templates/`.
Treat those templates, their paired `.properties.json` files, and the generated schema in `schemas/` as the core product.

## Source Of Truth

- Workflow templates live in `.github/workflow-templates/*.yml`.
- Each template must have a matching `.github/workflow-templates/<name>.properties.json` file.
- Reusable workflows live in `.github/workflows/reusable-*.yml` and are called by consumer repositories at the job level.
- The workflow-template metadata schema is generated by `scripts/update-workflow-template-properties-schema.mjs`.
- Schema fixtures live under `schemas/fixtures/workflow-template-properties/`.
- Repository automation lives under `.github/workflows/` and must not be confused with consumer-facing templates.

## Change Rules

- Do not hand-edit generated schema output when the generator is the right source to change.
- Do not patch generated, cache, build, or dependency output as a substitute for fixing source files.
- Preserve the `$default-branch` placeholder in starter workflow templates when the workflow should target the consumer repository default branch.
- Keep action references intentionally pinned. Existing templates prefer full commit SHAs with the upstream version in a comment.
- Keep workflow permissions least-privilege. Start from `contents: read` and grant write permissions only for jobs that need them.
- Keep workflow jobs bounded with `timeout-minutes` and useful `concurrency` groups.
- Keep secrets optional where practical, and document required secrets explicitly.
- Prefer SHA-pinned reusable workflow examples for cross-repository callers.
- Store temporary command output under `temp/`, not the repository root.

## Validation

Run the narrowest relevant checks first, then broaden before finalizing:

```bash
npm run typecheck
npm run schema:test:workflow-template-properties
npm run lint
```

When schema generation logic, schema docs, or template metadata changes, also run:

```bash
npm run schema:update:workflow-template-properties
npm run schema:check:workflow-template-properties
```

`schema:update:workflow-template-properties` fetches upstream category and icon data, so it can fail when network access is unavailable.

## Documentation Expectations

- Keep `README.md` focused on discovery, template inventory, and quick start.
- Keep `USAGE.md` focused on per-template setup and troubleshooting.
- Put longer how-to material under `docs/guides/`.
- When direct-copy instructions are shown, copy workflow YAML into the consumer repository `.github/workflows/` directory. `.properties.json` metadata belongs in a template repository, not a normal consumer workflow directory.
